强化学习在机器人控制中表现出较强的决策能力与环境适应性。
但在实际部署中，训练平台与真实机器人环境仍存在明显差异。且现有方法往往依赖高并行训练，而Webots等主流仿真平台难以直接支持大规模并行。
因此，有必要构建兼具安全验证能力与工程可迁移性的中间验证平台，以缩小训练环境与真实系统之间的差距。

同时，安全验证过程需要在条件一致、流程可复现的实验设置下开展，以保证不同算法与控制模块在统一评价基准上的横向可比性，并减少由环境差异带来的评估偏差。因此，本章重点关注统一实验环境与标准化评估流程的构建，使不同控制模块和安全策略能够在一致条件下进行比较。

前两章分别从约束流形安全过滤和多机器人几何约束策略学习两个角度提出了安全强化学习方法，相关实验主要在 \textit{Safety-Gymnasium} 中独立开展，缺少在统一仿真环境下对两类方法进行端到端运行与公平比较的条件。

基于上述需求，本章设计并实现了一套支持远程访问的容器化仿真验证平台。
该平台基于开源教学系统 \textit{RoboticsAcademy} 进行扩展，由面向多机器人编队任务的场景层、容器化算法层以及可复现评估管线组成，能够在隔离运行环境中承载不同安全强化学习方法的协同执行与对比评估。

在执行管线层面，平台将前两章方法以
\[\text{MAPPO} \;\rightarrow\; \text{几何协调层（GCPL）} \;\rightarrow\; \text{硬安全投影层} \;\rightarrow\; \text{差速映射}\]
的形式串联接入，其中强化学习策略输出任务驱动的控制意图，几何协调层在群体层面进行几何一致性整理，硬安全层对控制输入施加约束投影，差速映射给出可执行控制指令。该串联仅用作 CMPL 与 GCPL 在平台上的统一执行接口，不构成独立的方法贡献。

本章通过轻量级的 \textit{pure-python} MVP 系统对上述执行链路进行功能性验证，并在 \textit{Webots} 高保真仿真环境中开展多场景实验评估，用以检验平台对 CMPL 与 GCPL 的端到端集成、公平比较与可复现评估的支撑能力。

\subsection{容器化与并行调度机制}

针对强化学习算法对高并行训练环境的需求，以及机器人仿真环境在部署与依赖管理方面的复杂性，平台采用基于 Docker 的容器化运行机制对实验环境进行统一封装。

容器化运行机制中，每个实验任务均运行于独立的容器实例中，容器内部集成仿真引擎（如 Gazebo、Webots）、ROS 2 中间件以及算法执行模块。
该设计使不同实验之间在软件依赖、运行状态及资源使用上实现完全隔离，从而避免环境冲突问题。
同时，容器镜像在构建阶段已固定所有依赖版本，确保实验在不同时间与不同设备上的一致性，从而提升实验结果的可复现性。

在原有平台镜像基础上，本平台进一步集成深度学习框架与安全强化学习相关组件，从而构建完整的算法运行环境。
通过在容器内引入 PyTorch 及其相关依赖，并挂载外部代码目录的方式，将 Safety-Gymnasium 环境与 safepo 算法库以可编辑形式集成至容器中，使算法模块能够直接调用对应接口，以支持策略网络的加载与推理。
同时，前述安全算法相关实现作为独立模块进行管理，并通过环境变量实现路径配置，从而保证系统结构的清晰性。

在此基础上，平台通过容器调度机制支持多实例并行运行。后端系统根据用户请求动态创建、启动或销毁容器，实现实验任务的按需分配与资源管理。
该机制不仅能够支持多用户并发访问，还可为强化学习算法提供并行环境，从而提升样本采集效率与训练速度。

容器化并行训练的数据链路如图~\ref{fig:chap5_parallel_orch} 所示：统一镜像在构建阶段预先固化运行依赖，并启动 $K$ 个相互隔离的容器环境，
各容器独立执行仿真与策略交互，采样轨迹数据集中送入 MAPPO 训练端完成参数更新，更新后的策略再分发回各容器，
从而形成并行采样、集中训练、异步更新的闭环流程。

\subsection{双通道可视化机制}

为提升机器人系统运行状态的可解释性与调试效率，平台设计了一种融合语义级状态表达与原生仿真界面展示的双通道可视化机制。
该可视化机制同时支持对算法状态的分析与对仿真场景的完整观测，从不同维度增强系统整体的可视化表达能力。

双通道并行的整体数据链路如图~\ref{fig:chap5_dual_vis} 所示。
语义级可视化通道（WebGUI）从仿真环境中提取机器人位姿、传感器数据以及安全约束等关键状态信息，并将其统一组织为结构化数据，通过 WebSocket 实时推送至前端。
前端采用组件化渲染方式，将上述数据映射为轨迹曲线、激光扫描、力向量以及约束边界等二维或抽象图形，从而直观呈现算法的内部状态与决策过程，使系统运行逻辑具备更强的可分析性与可解释性。

原生界面可视化通道（noVNC）则对仿真软件（如 Gazebo、RViz 或 Webots）的图形界面进行远程转发。
同源的仿真器图形界面在 Xvfb 承载下由 Xvnc 编码为 VNC 帧流，经 noVNC 客户端解码后以 iframe 嵌入前端页面，从而完整保留原生仿真工具的交互能力与视觉效果，使用户能够获得与本地运行一致的使用体验。

两条可视化通路在运行过程中相互独立，互不阻塞。用户在前端产生的交互指令通过反向控制链路传递至仿真器，实现从状态展示到控制执行的闭环过程。图~\ref{fig:chap5_dual_vis} 重点刻画了两类可视化路径在同一状态源基础上的异构表达关系及其运行边界。
统一的仿真状态源保证了不同可视化通路所呈现信息的一致性，避免了多系统并行渲染时可能出现的状态偏差问题；渲染过程的解耦设计则使语义级可视化能够以轻量方式高频刷新，不受远程桌面帧传输带宽的限制。该机制在实验过程中提供了稳定一致的观测基础。

\subsection{安全评估验证机制}
针对安全强化学习实验中复现难度较高、算法评价标准不统一的问题，平台设计了一种安全评估与模型验证机制，
通过统一的运行脚本，可以在同一环境中顺序执行多种算法配置，并对实验结果进行集中管理，从而实现对不同算法在一致条件下进行系统性比较。

该机制首先对实验环境与任务配置进行标准化描述，包括机器人模型、场景结构、障碍物分布以及传感器参数等信息。不同算法均在相同配置下运行，以减少外部环境差异对结果造成的影响。在此基础上，平台设置统一的指标体系，对算法表现进行多维度评估。
评估指标包含任务完成率、累计奖励，以及与安全相关的度量，例如约束违反次数、最小安全距离以及累计安全代价等，从而更加全面地反映算法的实际表现。

为支持已有模型的复现与对比，平台提供模型导入与验证接口。用户可以上传训练完成的策略模型文件，系统在加载模型后自动执行标准化验证流程，包括环境初始化、策略推理以及结果记录。
该机制使不同来源的算法能够在统一平台上进行直接对比，避免因实现差异带来的评估偏差。

安全评估与模型验证机制的整体流程如图~\ref{fig:chap5_safety_eval} 所示。标准化任务配置统一下发至验证流水线，内建算法与用户上传模型沿相同执行路径完成实验过程，从而避免由实现差异带来的评估偏差。
评估结果按照统一指标体系进行汇总与分析，并进一步生成跨算法对比报表。同时，系统支持表格结果、轨迹可视化以及学习曲线的自动生成与导出，实现实验结果的结构化呈现。

\subsection{多机器人实验场景构造}

\subsubsection{（1）差速 E-puck 机器人建模}

为保证仿真环境与前文方法建模的一致性，平台选用 Webots 内置的 E-puck 机器人模型作为基础载体，并保留其标准差速运动学参数。
机器人轮半径设为 $r_w = 0.0205~\text{m}$，轮间距为 $L = 0.052~\text{m}$，最大轮速为 $\omega_{\max} = 6.28~\text{rad/s}$，对应最大线速度约为 $v_{\max} \approx 0.129~\text{m/s}$。

机器人状态以二维平面位置 $(x, y)$ 与航向角 $\theta$ 描述，其速度表示为线速度 $v$ 与角速度 $\omega$。
差速驱动模型通过左右轮角速度 $(\omega_L, \omega_R)$ 与 $(v, \omega)$ 之间的映射关系进行刻画，即
\begin{equation}
v = \frac{r_w}{2}(\omega_R + \omega_L), \qquad 
\omega = \frac{r_w}{L}(\omega_R - \omega_L).
\end{equation}

该建模方式与第四章中的状态定义保持一致，从而确保策略在不同仿真环境中的表达具有一致的物理语义，为后续跨平台验证提供基础。

\subsubsection{（2）场景构建与参数化设计}

对于具体场景，本文基于 Webots 构建了用于多机器人编队导航与避障任务的标准实验环境。
场景空间为 $3~\text{m} \times 3~\text{m}$ 的二维平面，在 $y$ 轴方向设置两面静态障碍墙，其中心分别位于 $(-1, 0)$ 与 $(1, 0)$，从而形成一条具有约束宽度的通道结构。
目标点固定设置在 $(0, 2)$ 位置，用于引导机器人完成导航任务。

为增强实验的灵活性与可扩展性，场景参数采用动态配置方式进行管理。
机器人数量、初始分布以及编队形态（如直线、楔形、圆形及网格结构）均通过环境变量进行控制，无需修改底层场景文件即可实现任务切换。
这种参数化设计使得平台能够在统一框架下支持多种实验配置，提升了系统的复用性。

\subsubsection{（3）ROS2 多机器人通信接口设计}

为实现多机器人系统中状态信息与控制指令的高效传输，平台基于 ROS 2 构建统一的通信接口体系。
针对多智能体场景，系统采用规范化的话题命名方式，对每个机器人个体建立独立的通信通道，从而保证数据交互的清晰性与可扩展性。

机器人位姿与速度信息通过 \path{/agent_i/odom} 话题发布，控制指令通过 \path{/agent_i/cmd_vel} 下发；
在需要状态估计的场景中，可通过 \path{/agent_i/state_estimate} 话题传递融合后的位姿信息。此外，系统定义了全局编队状态话题，用于记录编队误差与协同指标的时序变化。

在控制执行层面，平台采用集中式调度策略。Supervisor 节点通过 Webots 提供的接口周期性获取所有机器人状态，并在统一控制周期内完成策略计算与控制指令分发。
从而有效降低多机器人协同控制中的通信复杂度，并保证各机器人动作在时间上的一致性。

\subsection{后端服务模块}

后端服务模块基于 Django 框架实现，主要负责业务逻辑处理、实验资源管理和任务运行调度。系统采用 RESTful 架构向前端提供统一的数据访问接口，使前端页面与后端业务逻辑保持相对独立，便于后续功能扩展和模块维护。

从功能划分来看，后端服务主要包括实验任务管理、API 接口服务、数据库管理以及系统调度与控制四个部分。

\subsubsection{（1）实验任务管理}

实验任务管理模块用于维护平台中的机器人实验任务及其配置数据。每个实验任务都包含相对完整的环境描述信息，例如机器人模型、仿真场景、实验工具和算法模板等。后端将这些信息按照统一的数据结构进行组织，使不同类型的实验任务能够在同一平台中被检索、加载和调度。

% 当用户在前端选择某个实验任务后，后端会根据任务标识查询数据库，读取与该任务对应的配置信息，并将结果返回给前端。前端再依据这些配置加载相应的仿真环境、参数信息和界面组件，从而完成实验初始化。通过这种方式，任务配置与前端显示逻辑相互分离，平台在增加新任务时不需要大幅修改前端页面结构。

\subsubsection{（2）API 服务提供}

为实现前后端之间的数据交互，平台设计了一组 REST API 接口，用于提供实验任务查询、实验配置加载和代码模板下载等服务。前端应用在不同操作阶段通过 HTTP 请求调用对应接口，后端则根据请求内容完成数据查询、权限校验和结果返回。

% 例如，用户进入平台后，前端会先调用任务列表接口，从后端获取当前可用的实验任务；当用户进一步选择某个任务时，系统再调用实验配置接口，加载该任务对应的仿真环境、机器人模型和默认算法模板。基于 REST API 的通信方式降低了页面逻辑与后端实现之间的耦合程度，也使后续增加新的任务类型或评估接口更加方便。

\subsubsection{（3）数据库管理}

平台采用 PostgreSQL 作为后台数据库，用于存储实验任务、仿真环境配置、机器人模型和辅助工具等核心数据。
% PostgreSQL 在稳定性、扩展性和复杂查询支持方面较为成熟，能够满足机器人实验平台对结构化数据管理的需求。

在数据库设计中，平台主要包含以下几类核心数据对象：
\begin{itemize}
    \item Task：具体的机器人实验任务；
    \item Universe：实验任务对应的仿真环境配置；
    \item World：具体的仿真场景；
    \item Robot：机器人模型及其控制接口；
    \item Tool：实验过程中使用的辅助工具组件。
\end{itemize}

通过上述数据对象，平台可以对实验任务、场景资源和机器人配置进行统一管理，并支持不同仿真环境之间的组合与复用。
% Django 提供的对象关系映射（ORM）机制也简化了数据库访问过程，开发人员可以通过 Python 对象完成数据查询和更新，减少直接编写 SQL 带来的维护成本。

\subsubsection{（4）系统调度与控制}

系统调度模块负责管理实验任务从启动到结束的完整生命周期。当用户提交实验运行请求后，后端会根据任务配置启动对应的容器实例，并建立前端、算法进程和仿真环境之间的通信连接。在实验执行过程中，后端持续维护任务运行状态，支持启动、暂停、终止等控制操作，同时记录容器和任务进程的基本状态信息。该模块保证了实验执行流程的有序性，也为运行监控、异常排查和结果记录提供了基础。

通过上述模块设计，后端服务系统实现了机器人实验任务管理、数据服务以及仿真调度等核心功能，为平台提供了稳定可靠的运行基础。同时，基于 RESTful 架构与模块化设计思想，使系统具有良好的可扩展性，可以方便地接入新的机器人实验任务与仿真环境，从而满足多机器人强化学习研究的需求。

\section{功能验证}
\label{sec:chap5_mvp}

为验证平台所集成的分层控制栈（MAPPO 策略层、第四章 GCPL 几何协调层、第三章 CMPL 安全投影层）在无仿真器依赖的条件下即可协同工作，
本节在 Python 环境中实现了一个最小可行演示（Minimum Viable Demo，MVP）。MVP 与第五章后续 Webots 高保真验证共用同一分层控制代码，
仅将仿真环境从 Webots 替换为基于 Safety\-Gymnasium 的纯 Python 仿真，目的是在不引入任何图形化环境的情况下检验软硬交接接口的正确性与策略加载链路的完整性。

\subsection{验证场景与评估指标}
\label{sec:chap5_mvp_setup}

验证场景沿用第四章的 \texttt{SafetyPointMultiFormationGoal0-v0} 任务，3 个点智能体以楔形队列出发，穿越两堵 Sigwall 之间的狭窄通道抵达 $(0, 2)$ 处的目标区域。
量化指标分别为最小两两距离 $\min_t \min_{i\neq j}\|p_i(t)-p_j(t)\|$、编队误差 $\frac{1}{N(N-1)/2}\sum_{i<j}\bigl|\|p_i-p_j\|-d^*\bigr|$（其中 $d^*=0.5$~m 为目标间距），
以及目标到达数 $\#\{i:\|p_i^{end}-p_{\text{goal}}\|<0.3~\text{m}\}$。

\subsection{三模式对比}
\label{sec:chap5_mvp_3mode}

通过开关 GCPL\_RL\_WEIGHT 和 GCPL\_ENABLE\_CMPL 两个环境变量，MVP 分别运行三种模式：
MAPPO\_only（仅 RL 策略）、GCPL\_only（仅 GCPL 几何驱动）、fusion（完整分层栈）。
图~\ref{fig:chap5_mvp_3mode} 展示了三种模式下 3 个智能体在 10 秒仿真时间内的世界坐标轨迹。

\begin{figure}[htbp]
    \centering
    \includegraphics[width=0.98\linewidth]{figs/chap5_mvp_3mode.pdf}
    \caption{MVP 验证实验}
    \label{fig:chap5_mvp_3mode}
\end{figure}

MAPPO\_only 模式中，关闭 GCPL 与 CMPL 后，欠训练的 MAPPO 策略（seed\_002, $\mathrm{EpRet}\approx 3.4$）产生的控制输出使三机器人呈发散趋势运动，尽管其质心在轨迹中偶然经过目标区域并被判定为 reached=True，但执行过程中累计出现 33 个时间步的机器人间硬约束违反（$\min d_\mathrm{pair}=0.043$~m，显著低于 $R_\mathrm{hard}=0.25$~m）。该现象表明，在缺乏安全机制约束的情况下，即便任务指标表面达成，系统仍可能伴随严重的安全风险。

GCPL\_only 模式中，仅依赖几何约束机制进行控制。此时，GCPL 通过几何力建模与硬约束投影有效维持了编队结构（$\min d_\mathrm{pair}=0.290$~m，全程无碰撞），但由于缺乏强化学习提供的目标驱动信息，系统始终停留于初始区域附近，在 600 步内未能形成有效推进，表现出明显的任务不可达性。

fusion 模式下，欠训练策略输出经由几何协调层的平滑处理以及硬安全层的约束修正后，系统在整个运行过程中始终满足安全约束（$\min d_\mathrm{pair}=0.290$~m，零碰撞）。然而，由于策略本身存在偏置，整体运动方向出现系统性漂移，导致机器人群体向左侧障碍物中心 $(-1,0)$ 偏移（$\min d_\mathrm{wall}=0.378$~m，仍高于硬约束半径 $0.35$~m），最终未能穿越通道。该结果进一步说明，安全机制能够在策略尚未充分收敛时提供有效保护，但任务完成性能仍依赖于策略本身的优化程度。

综上，MVP 实验验证了分层控制栈在纯 Python 环境中的接口一致性与融合逻辑正确性。安全层在其作用范围内稳定维持硬约束条件：fusion 与 GCPL\_only 模式下均未发生碰撞，而 MAPPO\_only 模式中的 33 次碰撞则直接源于安全模块的显式移除。这一结果为后续在 Webots 高保真环境中的系统验证提供了可靠的实现基础。

\section{Webots 容器化仿真实验}
\label{sec:chap5_webots_eval}

\end{itemize}

\section{本章小结}

本章针对前两章所提方法缺乏统一仿真环境进行端到端运行与公平比较的问题，构建了一套面向多机器人安全强化学习的容器化仿真验证平台。围绕实验复现性、并行支持以及安全评估等方面进行系统架构与核心运行机制设计。

整体设计采用分层结构，对仿真环境、通信中间件、容器化运行环境、算法执行逻辑以及前端交互界面进行解耦，
并通过容器化运行方式使实验能够在隔离环境中稳定执行，同时支持多实例任务的并行调度，从而提升资源利用效率与实验吞吐能力。
双通道可视化使系统运行过程更具可解释性，实时通信机制则维持低延迟的数据交互。
平台具体实现覆盖仿真执行、前端交互、后端服务与通信链路等核心模块，形成从任务配置、环境启动到运行监控与结果分析的完整流程。统一的数据接口与调度逻辑使各功能单元能够高效协同，从而支撑多场景、多任务条件下的机器人算法验证。

平台为 CMPL 与 GCPL 的端到端运行提供了统一的实验载体与标准化评估流程，便于在相同任务条件下开展公平比较。
在\ref{sec:chap5_mvp} 的 MVP 实验与\ref{sec:chap5_webots_eval} 的 Webots 仿真实验中，
四档消融配置在共 12 次随机种子测试下均未出现碰撞行为，并清晰呈现了 CMPL 与 GCPL 在不同权重组合下的协同特性。
这表明，平台能够在统一仿真环境下稳定执行 CMPL 与 GCPL 的串联控制管线，并在不同消融配置之间给出一致、可复现的端到端实验结果。
